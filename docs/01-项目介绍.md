## 一、更名说明

本项目由原 **DAO（道）** 更名为 **SunDAO（深道）**。

- **DAO（道）**：项目名称源于道家哲学"道生一，一生二，二生三，三生万物"，体现 Being + Type + Relation 的核心三元组设计哲学。
- **SunDAO（深道）**："深"寓意深度（Deep）、深耕（Deep-rooted）、深智（Deep Intelligence），与项目从数据基座向世界模型驱动智能基础设施的演进方向一致。

更名后，原 dao 单仓架构拆分为多子项目独立仓库，SunDAO 作为总项目统一协调。原 dao 项目的历史设计文档、架构决策记录仍具参考价值，已归档保留。

## 二、项目定位

**深道（SunDAO）是世界模型驱动的智能基础设施。**

为 AI Agent 提供可持久化、可溯源、可推理、可审计的语义数据基座；  
为企业与开发者提供"本体驱动、多端统一"的智能应用接入平台；  
为数据主权敏感行业提供私有化部署 + 国密合规的自主可控方案。

**核心范式转换**：

```
传统范式：                    SunDAO 范式：
智能 = 模型权重               智能 = 世界模型（数据结构 + 规则 + 关系）+ 模型（生成能力）
数据 = 被动输入               数据 = 主动语义（自带行为定义）
学习 = 梯度下降（$10^6/次）    学习 = 写入新概念（$0/次）
记忆 = 上下文窗口（易失）      记忆 = 推理缓存持久化为 Being（跨会话、跨设备、终身）
审计 = 外部日志（可篡改）      审计 = WAL 原生（物理不可篡改）
```

**五大核心特征**：

- 🌍 **世界模型驱动** — Being/Type/Relation 构成可自我演化的显式世界模型
- 🧠 **终身记忆系统** — 推理缓存持久化为 Being，跨会话、跨设备、终身继承
- 🔍 **白箱推理** — 推理链 Being 记录完整推导路径，可审计可质疑
- 🔐 **内生审计** — 推理与审计的同一 WAL 事务，物理不可篡改
- 🚀 **水平扩展** — 原生分布式集群（分片路由 + 节点发现 + 共识机制），单机到百节点无缝扩展
- 🛡️ **国密合规** — SM2/SM3/SM4 贯穿存储到传输，等保三级/EU AI Act 原生满足

> **我们不是另一个数据库，而是 AI 与世界的"认知接口"。**
> **我们不只是基础设施，而是让数据自动变成智能的平台。**
> **我们不只是存储数据，而是在存储一个可以自我演化的世界模型。**

## 三、范式革命：从"模型中心"到"世界模型中心"

### 3.1 隐式范式的危机

当前所有主流 LLM 共享同一个底层假设：**智能存在于神经网络的权重中**。这导致了五大结构性缺陷：

1. **幻觉不可解释** — 知识是权重空间的统计涌现，无法展示"为什么知道"
2. **知识无法精确修正** — 只能通过大规模微调间接"敲打"权重
3. **推理不学习** — 每次对话结束后，模型不会记住用户教它的任何东西
4. **不可审计** — 推理过程是黑箱，无法展示推导路径
5. **上下文瓶颈** — 128K-1M token 的窗口对于终身记忆杯水车薪

### 3.2 显式范式的核心主张

SunDAO 将智能的载体从"神经网络权重"转移到"数据结构"：

| 维度 | 隐式范式（训练 LLM） | 显式范式（世界模型） |
|------|---------------------|---------------------|
| **知识载体** | 神经网络权重 | Being/Type/Relation |
| **知识更新** | 重训练/SFT/RLHF（贵、慢） | `write()` / `relate()`（即时、几乎零成本） |
| **推理过程** | 黑箱 | 白箱（推理链 Being） |
| **错误修正** | 收集反馈→调参→hope for the best | 直接修改 Being |
| **记忆机制** | 上下文窗口（易失） | KV Cache 持久化（终身） |
| **可审计性** | 外部日志（可篡改） | WAL 原生（物理不可篡改） |

### 3.3 双系统分工

借用认知科学的双系统理论：

```
System 1（快思考）—— V4 神经网络
├── 负责：语言生成、语义编码、创造性联想
└── 不修改权重，每次推理时接收 System 2 注入的记忆

System 2（慢思考）—— SunDAO 外显世界模型
├── 负责：事实存储、规则执行、逻辑推理、个性化适应
└── 可查询、可更新、可反思、可审计
```

**关键洞察**：不修改 System 1 的权重，而是让 System 2 在每次推理时动态注入相关知识。

## 四、三大核心特征

### 特征一：🌍 世界模型驱动（World Model Driven）

**核心理念**：世界模型不在神经网络权重中，而在数据结构（Being/Type/Relation/Contract/Version）中。LLM 只是查询和操作这个世界模型的工具。

**技术实现**：
- **Being**：一切可寻址实体，固定记录设计，数十个通用业务字段
- **Type**：自举类型系统，运行时热注册，无需重启
- **Relation**：关系即 Being，精简记录设计，图遍历 O(1) 每跳
- **Contract**：业务规则脚本在写入路径执行，物理上不可绕过
- **Version**：多版本版本链，immutable-append，时间旅行查询

### 特征二：🧠 终身记忆与白箱推理（Lifelong Memory & White-box Reasoning）

**核心理念**：AI 不再"失忆"——每次对话的推理缓存持久化为 Being，跨会话、跨设备、终身继承。

```
Layer 7: 元记忆 (Meta)        — 关于记忆的记忆
Layer 6: 程序记忆 (Procedural) — 技能/模式/工作流
Layer 5: 语义记忆 (Semantic)   — 知识/概念/事实
Layer 4: 情节记忆 (Episodic)   — 事件/经历/对话历史
Layer 3: 工作记忆 (Working)    — 会话上下文/短期缓存
Layer 2: 感知记忆 (Sensory)    — 原始输入/瞬时感知
Layer 1: 感官输入 (Input)      — 多模态原始数据
```

**白箱推理**：
- 每次推理生成推理链 Being，记录完整推导路径
- 用户可追问"为什么这么回答？"——AI 展示完整推理链
- 推理与审计的同一 WAL 事务，物理不可篡改

### 特征三：🔐 内生审计与合规（Intrinsic Audit & Compliance）

**核心理念**：审计不是附加功能，是存储层的内生属性。

| 法规 | 要求 | SunDAO 满足方式 |
|------|------|----------------|
| **等保三级** | 国密算法、审计日志 | SM2/SM3/SM4 + WAL 不可篡改链 |
| **EU AI Act**（2026.8 生效） | 决策可追溯 | 推理链 Being + 完整推理路径 |
| **GDPR Art.30** | 处理活动记录 | 审计日志记录所有读写操作 |
| **SOX** | 财务数据不可篡改 | immutable-append + 链式哈希 |

**五层保障**：
1. **私有化部署可选**：数据不出域，支持单机到集群部署，满足金融/医疗合规
2. **集群高可用**：分片路由 + 节点发现 + 共识机制，自动故障切换，支撑 SaaS 规模化
3. **immutable-append 审计**：WAL 即审计日志，链式哈希物理不可篡改
4. **国密合规**：SM2/SM3/SM4 贯穿存储到传输，原生设计非后期插件
5. **推理链白箱**：推理链 Being 记录完整推导路径

- Agent 是可选增强层，不是 App 的必需依赖

### 5.2 SunDaoQL — 世界模型的物理载体

**一句话定位**：多模态存储引擎，图+列+向量+全文统一内核，零外部数据库依赖。

**核心能力**：
1. **ACID 图引擎**：精简关系记录，双向邻接表，图遍历 O(1) 每跳
2. **列式存储**：固定 Being 记录，批量扫描，分析型查询性能优异
3. **向量搜索**：原生向量索引加速，支持高维 Embedding，通过行业标准基准测试
4. **全文索引**：内置倒排索引，无需 ElasticSearch
5. **审计级日志**：append-only 日志，国密算法贯穿存储全链路
6. **分布式就绪**：分片路由 + 共识机制 + 副本复制，单机到百节点无缝扩展

**竞品对比**：

| 竞品 | 类型 | SunDaoQL 的核心差异 |
|------|------|---------------------|
| **Neo4j** | 图数据库 | Neo4j 仅图存储；SunDaoQL 同一进程内集成图+列+向量+全文，无需多库组合 |
| **Pinecone / Milvus** | 向量数据库 | Pinecone 仅向量；SunDaoQL 向量搜索与图遍历可在同一事务中执行 |
| **ClickHouse** | 列数据库 | ClickHouse 仅列存储；SunDaoQL 列存储与图关系在同一查询中联合分析 |
| **TiKV + TiFlash** | 混合存储 | TiKV/TiFlash 是分离进程/集群；SunDaoQL 是单一进程，部署复杂度低一个数量级 |
| **Dgraph** | 原生分布式图 | Dgraph 不支持向量搜索和全文索引；SunDaoQL 四模态统一 |
| **PostgreSQL + 插件** | 关系型 | PG 的图/向量/全文依赖插件，数据模型不匹配；SunDaoQL 原生语义数据模型 |

**独立产品价值**：✅ **可作为"新一代多模数据库"单独销售**
- **目标客户**：对数据架构简化有强需求的中小企业、IoT 平台、AI 基础设施团队
- **定价参考**：¥15-50 万/节点/年（私有化授权）
- **核心卖点**：用一个进程 + 一个数据模型替代 Neo4j + Pinecone + ClickHouse + ES 的组合，运维成本降低 80%
- **使用场景**：知识图谱、RAG 向量库、时序分析、全文检索的融合场景

**与其它子项目的组合价值**：
- **+ SunDaoQLServer**：通过网络暴露 QL 能力，支撑 SaaS 多租户模式；QLServer 是 QL 的"网络接口"
- **+ SunDaoAgent**：GraphMemory 直接在存储层实现，Agent 与数据零距离，无需 ETL 胶水代码
- **+ SunDaoLLM**：Embedding 生成后直接写入同一进程内存，无网络序列化和跨进程拷贝开销
- **组合增值**：当四者组合时，AI 的记忆、推理、数据查询全部在同一技术栈内完成，端到端延迟 < 10ms

### 5.3 SunDaoQLServer — 世界模型的网络接口

**一句话定位**：多模数据库的云服务层，DSL/Meta/Auth 三合一，支持单机到百节点集群。

**核心能力**：
1. **数据服务层**：查询语言服务，RESTful + WebSocket 双协议
2. **元数据服务层**：Being 元信息管理、类型注册、Schema 演进、权限校验
3. **Auth & Multi-tenancy**：RBAC 细粒度权限、资源配额、租户数据隔离
4. **集群协调器**：分片路由 + 节点发现 + 共识机制 + 副本复制
5. **多部署形态**：单机 / 主从 / 集群 / SaaS，同一套二进制

**竞品对比**：

| 竞品 | 类型 | SunDaoQLServer 的核心差异 |
|------|------|---------------------------|
| **Hasura** | GraphQL 引擎 | Hasura 面向关系型 PostgreSQL；QLServer 面向语义级图+向量+列多模态 |
| **Supabase** | BaaS | Supabase 后端是 PostgreSQL；QLServer 后端是图+向量+列，面向 AI 场景 |
| **Firebase** | 移动 BaaS | Firebase 云端托管不可私有化；QLServer 支持完全私有化部署 |
| **TiDB** | 分布式 SQL | TiDB 是关系型 HTAP；QLServer 支持图遍历和向量搜索，TiDB 不支持 |
| **CockroachDB** | 分布式 SQL | CockroachDB 面向金融事务；QLServer 面向 AI + 语义数据 |
| **Neo4j Aura** | 图数据库云 | Aura 仅图存储；QLServer 四模态 + 国密 + 私有化，合规性更强 |

**独立产品价值**：✅ **核心变现载体，可单独运营 SaaS**
- **目标客户**：需要后端服务的开发者、中小企业、需要私有化数据服务的大企业
- **定价模式**：Free / Starter ¥500/月 / Pro ¥2,000/月 / Business ¥5,000/月 / Enterprise 定制
- **核心卖点**：部署一个二进制文件，获得图+向量+列+全文+审计+国密的完整后端服务
- **使用场景**：AI 应用后端、知识管理平台、可审计数据服务、国密合规数据底座

**与其它子项目的组合价值**：
- **+ SunDaoQL**：QLServer 是 QL 的网络化封装；无 QL 则 QLServer 无法独立运行（强耦合）
- **+ SunDaoAgent**：Agent 通过元数据服务接入，AI 层与数据层解耦，Agent 不感知存储细节
- **+ SunDaoApp**：App 通过 WebSocket 连接 QLServer，实现多端数据的实时同步与冲突解决
- **+ SunDaoLLM**：QLServer 调用 LLM 进行自然语言→DAO-QL 转换、Embedding 实时生成
- **组合增值**：QLServer + App = 基础数据交互平台；+ Agent = 完整的"可审计 AI 助手"

### 5.4 SunDaoAgent — 世界模型的操作接口

**一句话定位**：企业级 AI Agent 框架，GraphMemory + MCP + 白箱推理，让 AI 真正理解业务数据结构。

**核心能力**：
1. **GraphMemory**：基于图结构的情境化记忆，子图查询 O(1) 每跳，支持 7 层记忆模型
2. **MCP 协议**：Model Context Protocol，标准化 AI 工具调用，可对接任意 MCP Server
3. **Sub-Agent**：分层多 Agent 协作，复杂任务分解为可并行的子任务
4. **Evolution**：AI 自主发现新概念、新关系，世界模型随使用自我演化
5. **白箱推理**：推理链 Being 记录完整推导路径，每次推理可审计可质疑
6. **架构解耦**：数据服务已剥离，Agent 专注 AI 计算层（元数据 + AI）

**竞品对比**：

| 竞品 | 类型 | SunDaoAgent 的核心差异 |
|------|------|------------------------|
| **LangChain** | Agent 框架 | LangChain 是通用胶水框架；SunDaoAgent 与存储深度集成，GraphMemory 原生而非外部插件 |
| **AutoGPT** | 自主 Agent | AutoGPT 易发散、难审计；SunDaoAgent 结构化记忆 + 白箱推理，可控可审计 |
| **CrewAI** | 多 Agent 协作 | CrewAI 基于纯文本协作；SunDaoAgent 基于图记忆的多 Agent 共享结构化知识 |
| **Dify** | 低代码 Agent | Dify 面向快速搭建（低代码）；SunDaoAgent 面向企业深度定制（代码级灵活） |
| **Coze** | 商业 Agent 平台 | Coze 是云端 SaaS，数据出域；SunDaoAgent 可完全私有化，数据主权可控 |
| **Microsoft Copilot Studio** | 企业 Agent | Copilot 绑定微软生态；SunDaoAgent 独立中立，国密合规 |

**独立产品价值**：✅ **可作为"企业级 Agent 框架"单独销售**
- **目标客户**：需要私有化 AI Agent 的中大型企业（金融、医疗、制造、政务）
- **定价参考**：¥10-30 万/年（私有化授权）
- **核心卖点**：AI 不仅"能聊天"，还能"理解业务数据结构"——知道客户 A 与订单 B 的关系，推理过程白箱可审计
- **使用场景**：智能客服（继承客户全历史）、合规审计助手（推理过程可追溯）、智能制造排程（理解设备-工单-物料关系）

**与其它子项目的组合价值**：
- **+ SunDaoQLServer**：Agent 通过 Meta 端口获取世界模型数据，零胶水代码接入业务数据
- **+ SunDaoLLM**：Agent 调用 LLM 进行推理，但记忆持久化在 SunDaoQL 中，实现"终身记忆"
- **+ SunDaoApp**：Agent 为 App 提供可选的 AI 增强能力；App 不依赖 Agent 即可直接连接 QL/QLServer 完成数据交互
- **组合增值**：Agent + LLM + QLServer = AI 拥有"长期记忆 + 推理能力 + 数据访问"，这是当前市面上任何单一产品都无法提供的组合

### 5.5 SunDaoLLM — 世界模型的推理服务平台

**一句话定位**：SunDAO 生态的推理服务平台，为所有子项目提供 AI 推理能力接入，按 Token 和订阅计费。

**核心能力**：
1. **LLM 推理 API**：提供标准化的推理接口，支持多模型后端，为 Agent 提供对话与推理能力
2. **推理缓存持久化**：跨会话推理加速，Agent 拥有"长期记忆缓存"
3. **Embedding 服务**：语义向量实时生成，支撑 RAG 检索和向量搜索
4. **NL→DSL 转换**：自然语言转结构化查询，降低开发者使用门槛
5. **国密传输**：国密算法贯穿推理服务全链路，满足合规要求

**竞品对比**：

| 竞品 | 类型 | SunDaoLLM 的核心差异 |

### 5.9 子项目关系网络

#### 依赖关系图

```
SunDaoQL (存储基座)
    ↑
    ├──→ 嵌入式直连 → SunDaoApp（本地模式，零网络依赖）
    │
    └── SunDaoQLServer（服务层）
            ↑
            ├──→ SunDaoApp（网络模式）
            ├──→ SunDaoQLWeb
            └──→ SunDaoAgent（AI 增强层 — 可选）
                    ↑
                SunDaoLLM
```

**依赖规则**：
- **QL → 无依赖**：最底层，所有子项目的最终数据底座
- **QLServer → 依赖 QL**：QLServer 启动时必须连接 QL 实例（本地或远程）
- **Agent → 依赖 QLServer**：Agent 通过元数据服务获取数据和权限，不直接操作存储层
- **LLM → 无硬性依赖**：可独立运行，但被 Agent（推理）和 QLServer（Embedding/NL→DSL）调用
- **App → 可直接依赖 QL（嵌入式）或 QLServer（网络模式）**：Agent 是可选增强，不是必需依赖
- **QLWeb → 依赖 QLServer**：通过 RESTful API 管理 QLServer 中的所有数据

#### 数据流向

```
用户输入 → SunDaoApp
              │
              ├──→ 嵌入式模式 → SunDaoQL（本地存储，零网络依赖）
              │
              └──→ 服务模式 → SunDaoQLServer → SunDaoQL
                            │
                            ├──→ AI 增强（可选）→ SunDaoAgent → SunDaoLLM
                            │
                            └──→ 数据查询 → SunDaoQL

开发者 → SunDaoQLWeb → SunDaoQLServer → SunDaoQL
```

#### 职责边界（清晰分离，不重叠）

| 职责 | 归属子项目 | 不归属（由谁负责） |
|------|-----------|-------------------|
| 物理存储（图/列/向量/全文） | SunDaoQL | QLServer/Agent 不直接操作存储文件 |
| 网络接口 / 多租户 / 集群 | SunDaoQLServer | QL 无网络能力，Agent 不处理租户隔离 |
| AI 推理 / 记忆 / 白箱逻辑 | SunDaoAgent | QLServer 不执行 AI 计算，QL 不感知 AI |
| LLM 推理 / Embedding 生成 | SunDaoLLM | Agent 调用 LLM 但不实现推理引擎 |
| UI 渲染 / 离线 / 跨平台 | SunDaoApp | Agent 不处理界面，QLServer 不处理客户端 |
| 调试可视化 / 集群监控 | SunDaoQLWeb | App 不面向开发者，Agent 不提供管理界面 |

## 六、从 DAO 到 SunDAO 的架构演进

### 第一阶段：DAO（多存储适配器架构）

原 DAO 项目采用 **8 存储组件** 架构：

```
PostgreSQL + MongoDB + Redis + Neo4j + ClickHouse + Qdrant + ElasticSearch + MinIO
```

每个组件解决特定问题，但组合后产生了远超 8 倍的复杂度：CDC 同步、Saga 补偿、查询路由、降级熔断等多套胶水代码。

### 第二阶段：DaoQL（统一存储内核）

从 8 组件走向 **单一自研存储内核**：

```
ACID 图引擎 + 列引擎 + 向量索引 + 全文索引 = 单一自研内核，零外部依赖
```

DaoQL 的核心洞察：**"需要建索引的字段都在图上和向量索引上，不需要建索引的都在列库上。"**

### 第三阶段：SunDAO（世界模型驱动智能基础设施）

将 DaoQL 单仓按职责拆分为 6 个独立子项目，形成世界模型驱动的智能基础设施：

| 演进方向 | 在世界模型中的角色 | 子项目 |
|----------|-------------------|--------|
| **存储层独立** | 世界模型的物理载体 | SunDaoQL — 单一自研内核，四模态存储 |
| **服务层独立** | 世界模型的网络接口 | SunDaoQLServer — DSL/Meta/Auth，多部署形态 |
| **AI 层独立** | 世界模型的操作接口 | SunDaoAgent — GraphMemory + MCP + 白箱推理 |
| **推理层独立** | 世界模型的推理服务平台 | SunDaoLLM — 推理 API，Token 计费 |
| **客户端独立** | 世界模型的可视化呈现 | SunDaoApp — 瘦客户端，离线渲染 |
| **工具层独立** | 世界模型的调试窗口 | SunDaoQLWeb — 可视化调试，集群监控 |

## 七、典型应用场景

### 场景 1：企业级可审计 AI 助手

**问题**：企业 AI 项目"失忆"且"黑箱"，无法用于关键决策，合规受阻

**SunDAO 方案**：
- 客户记忆图谱：客户 A→咨询产品 X→投诉订单 Y→客服 B 服务
- 白箱推理：AI 回答"推荐产品 Z"时，展示完整推理链（从哪些记忆推导出来）
- 内生审计：每次推理与审计记录同一 WAL 事务，等保三级/EU AI Act 天然合规
- 终身记忆：客服 B 离职后，新客服 C 继承全部客户记忆图谱

**涉及子项目**：SunDaoQL（存储记忆）+ SunDaoQLServer（网络服务）+ SunDaoApp（客服界面）；可选增强：SunDaoAgent（AI 推理）+ SunDaoLLM（推理服务）

### 场景 2：智能制造世界模型

**问题**：生产排程复杂，多系统数据割裂，AI 无法理解完整制造语境

**SunDAO 方案**：
- 设备 Being（状态/产能/维护）、工单 Being（优先级/交期/工艺）、物料 Being（库存/供应商/质量）
- 世界模型自动演化：新设备接入 → 自动建立 Type → 自动关联现有 Relation
- OR-Tools CP-SAT 排程优化 + 知识图谱工艺路线推荐 + LLM 自然语言指令理解
- 排程效率提升 40%，交期达成率从 75% 提升到 92%

**涉及子项目**：SunDaoQL（制造知识图谱）+ SunDaoQLServer（工厂数据服务）+ SunDaoApp（车间看板）；可选增强：SunDaoAgent（排程 Agent）+ SunDaoLLM（自然语言理解）

### 场景 3：多端统一的智能应用接入

**问题**：企业应用需要覆盖 PC、手机、平板、Web 等多端，数据同步复杂，体验割裂

**SunDAO 方案**：
- 统一世界模型后端：所有终端通过 WebSocket 连接同一 QLServer 集群
- 瘦客户端架构：SunDaoApp 本地仅缓存 UI 模板，业务数据实时同步
- 离线渲染能力：网络中断时仍可浏览界面，恢复后自动重放变更
- 跨平台覆盖：Windows / macOS / Linux / iOS / Android / Web

**涉及子项目**：SunDaoQLServer（统一后端）+ SunDaoApp（多端客户端）+ SunDaoQLWeb（管理后台）

> **远期愿景**：基于世界模型的语义标准，逐步支持去中心化网络与跨组织 AI 协作。
