## 一、背景与目标

### 1.1 现有业务基础

| 指标 | 数值 |
|------|------|
| 原有产品线 | 报价系统 / PLM / ERP / MES / WMS / AGV 调度 |
| 年销售收入 | **2,000 万人民币** |
| 客户数量 | 约 50-80 家中大型制造企业 |
| 行业分布 | 汽车零部件、电子制造、机械加工、新能源 |
| 销售模式 | 项目制 + 年度维护费 |
| 客单价 | ¥25-50 万/家（首年），¥5-10 万/家（维护费） |

### 1.2 核心洞察：现有客户的痛点就是 SunDAO 的机会

现有客户使用我们 6 个独立产品（报价/PLM/ERP/MES/WMS/AGV），本质上是在使用**6 个数据孤岛**：

```
报价系统 ──→ PLM ──→ ERP ──→ MES ──→ WMS ──→ AGV
   │           │         │        │        │       │
   └─ 数据不同步 ─┘         └─ 接口脆弱 ─┘    └─ 实时性差 ┘
   └─ 重复录入 ───┘         └─ 状态不一致 ─┘   └─ 无法追溯 ┘
```

**客户真实痛点**（来自现有业务反馈）：
1. **数据不通**：报价变更后，PLM、ERP、MES 数据不一致，生产错误频发
2. **追溯困难**：质量问题发生时，需要跨 5 个系统查记录，耗时 2-3 天
3. **AI 用不上**：各系统数据格式不统一，无法做全局 AI 分析
4. **运维负担重**：6 个系统 = 6 套数据库 = 6 套备份 = 6 套权限管理

**SunDAO 的替代价值**：
```
SunDAO 统一平台
    │
    ├── Being: 产品（报价信息 + BOM + 工艺路线）
    ├── Being: 订单（ERP 订单 + MES 工单 + WMS 出库单）
    ├── Being: 物料（PLM 物料 + ERP 库存 + WMS 仓位）
    ├── Relation: 产品→订单→工单→出库→AGV 搬运（全链路追溯）
    │
    └── AI Agent: "哪个订单的物料在仓库哪个位置？"
         └── 自然语言 → DSL → 跨系统查询 → 3 秒返回结果
```

## 三、分产品线转化路径

### 3.1 报价系统 → SunDAO 智能报价

**原有功能**：产品配置、价格计算、报价单生成、客户管理

**痛点**：
- 报价变更后，PLM/ERP 数据不自动同步
- 无法基于历史报价做智能推荐
- 客户询价响应慢（平均 2-3 天）

**SunDAO 升级方案**：

```
def Product {
    schema { name, specs, base_cost, margin_rate }
    template { 产品配置界面 }
    client_script { 价格自动计算 }
}

def Quote {
    schema { customer_id, product_id, quantity, price, expiry_date }
    template { 报价单界面 + PDF 生成 }
    client_script {
        // AI 智能推荐
        fn on_mount() {
            let similar = await $dao.query()
                .scanByDef("Quote")
                .filter({ customer_id: this.props.customer_id })
                .orderBy("created_at", "desc")
                .limit(10)
                .execute()
            this.state.suggested_price = ai_recommend_price(similar)
        }
    }
}

// 报价→BOM→工艺路线 自动关联
relation Quote → Product (HAS_BOM)
relation Product → ProcessRoute (HAS_PROCESS)
```

**转化话术**：
> "张总，您现在的报价系统改一个价格，PLM 和 ERP 里的数据要人工同步 2 次，平均每周出错 3 次。SunDAO 里报价、BOM、工艺路线是同一个 Being 的不同关系，改一次自动全链路同步。而且 AI 能根据历史报价自动推荐最优价格，响应时间从 2 天降到 2 分钟。"

**定价**：原有报价系统维护费 ¥3-5 万/年 → SunDAO 智能报价模块 ¥15-25 万/年

### 3.2 PLM → SunDAO 产品全生命周期管理

**原有功能**：BOM 管理、变更管理、文档管理、版本控制

**痛点**：
- BOM 变更后，ERP/MES 无法实时同步
- 文档分散在多个系统，查找困难
- 版本追溯依赖人工记录

**SunDAO 升级方案**：

```
def Product {
    schema { name, version, status, owner }
    // 版本链：每次变更自动生成新版本 Being
    // 历史版本通过 relation 关联，天然追溯
}

def BOM {
    schema { product_id, components[], quantity[] }
    // BOM 变更是 Being 的新版本，不是 UPDATE
    // 旧版本永远可查，审计天然合规
}

def ChangeRequest {
    schema { type, reason, impact_analysis, approver }
    client_script {
        fn on_submit() {
            // 自动评估变更影响范围
            let impact = $dao.query()
                .being(this.props.bom_id)
                .withRelations(["USED_IN_PRODUCT", "USED_IN_ORDER"])
                .execute()
            // 自动通知相关方
            this.$server.call({ method: "notify_stakeholders", params: { impact } })
        }
    }
}
```

**核心卖点**：
- **不可变版本链**：BOM 每次变更自动生成新版本，旧版本永远可查（传统 PLM 的"版本覆盖"问题彻底解决）
- **变更影响自动分析**：改一个零件，自动分析影响哪些产品、哪些订单、哪些工单
- **跨系统同步**：BOM 变更 → 自动推送 ERP/MES/WMS（基于 Being 变更订阅）

**定价**：原有 PLM 维护费 ¥5-8 万/年 → SunDAO PLM 模块 ¥25-40 万/年

### 3.3 ERP → SunDAO 企业资源管理

**原有功能**：采购、销售、库存、财务、生产计划

**痛点**：
- 库存数据与 WMS 不一致，盘点频繁出错
- 生产计划与 MES 脱节，排程靠 Excel
- 财务报表与业务数据不同步

**SunDAO 升级方案**：

```
def PurchaseOrder {
    schema { supplier_id, items[], total_amount, status }
    // 采购订单状态变化自动触发库存更新
    client_script {
        watch state.status(new_val, old_val) {
            if (new_val === "RECEIVED") {
                // 自动创建入库单
                $dao.mutate("create_inbound_order(...)")
                // 自动更新库存 Being
                $dao.mutate("update_inventory(...)")
            }
        }
    }
}

def Inventory {
    schema { material_id, quantity, location, batch_no }
    // 实时订阅：库存变化自动推送到 WMS/MES
}

// AI 智能排程
client_script {
    async fn generate_schedule() {
        let orders = await $dao.query().scanByDef("SalesOrder")
            .filter({ status: "CONFIRMED", delivery_date: { lte: "2026-06-30" } })
            .execute()
        // 调用 SunDaoAgent 的求解器工具
        let schedule = await $server.call({
            method: "solver_schedule",
            params: { orders, capacity: this.state.capacity }
        })
        this.state.schedule = schedule
    }
}
```

**核心卖点**：
- **实时库存**：ERP 库存与 WMS 仓位实时同步，零盘点差异
- **AI 智能排程**：自然语言输入"下周要交付 10 个订单，怎么排？"，Agent 自动生成最优排程
- **财务溯源**：每一笔财务数据可直接追溯到业务事件（Being 版本链）

**定价**：原有 ERP 维护费 ¥8-15 万/年 → SunDAO ERP 模块 ¥40-60 万/年

### 3.4 MES → SunDAO 制造执行系统

**原有功能**：工单管理、工序跟踪、质检记录、设备状态

**痛点**：
- 工单状态与 ERP/WMS 不同步
- 质检记录分散，追溯困难
- 设备故障响应慢

**SunDAO 升级方案**：

```
def WorkOrder {
    schema { product_id, quantity, priority, status }
    // 工单全生命周期：创建→下发→开工→报工→质检→入库
    // 每个状态变化都是 Being 的新版本，天然追溯
}

def QualityCheck {
    schema { work_order_id, inspector, result, defect_type }
    // 质检不合格 → 自动触发返工工单 + 通知质量经理
    client_script {
        watch state.result(new_val) {
            if (new_val === "FAIL") {
                $dao.mutate("create_rework_order(...)")
                $server.call({ method: "alert", params: { level: "HIGH" } })
            }
        }
    }
}

// 设备 Being：与 AGV 联动
def Equipment {
    schema { name, status, current_task, maintenance_due }
    // 设备故障 → 自动调度 AGV 搬运物料到备用设备
    client_script {
        watch state.status(new_val) {
            if (new_val === "DOWN") {
                $server.call({ method: "agv_reroute", params: { from: this.id } })
            }
        }
    }
}
```

**核心卖点**：
- **全链路追溯**：从原材料入库 → 工单 → 工序 → 质检 → 成品出库，一个 Being 关系链全追踪
- **质检 AI 辅助**：Agent 根据历史质检数据自动识别异常模式
- **设备预测维护**：基于 Being 时序数据预测设备故障

**定价**：原有 MES 维护费 ¥10-20 万/年 → SunDAO MES 模块 ¥50-80 万/年

### 3.5 WMS → SunDAO 智能仓储

**原有功能**：入库、出库、盘点、库位管理、批次管理

**痛点**：
- 库存数据与 ERP/MES 不同步
- 盘点效率低，差异大
- 无法支持 AGV 自动调度

**SunDAO 升级方案**：

```
def WarehouseLocation {
    schema { zone, shelf, level, capacity, current_being_id }
    // 库位 Being：实时记录存放的物料 Being
}

def InboundOrder {
    schema { supplier_id, items[], expected_date }
    // 到货 → 自动分配库位 → 通知 AGV 搬运
    client_script {
        fn on_arrival() {
            let locations = $dao.query()
                .scanByDef("WarehouseLocation")
                .filter({ zone: this.props.preferred_zone, capacity: { gt: 0 } })
                .execute()
            let best = optimize_location(locations, this.state.items)
            $server.call({ method: "agv_move", params: { target: best } })
        }
    }
}

// 自然语言查询："A 仓库里有多少批 2026 年 3 月入库的螺丝？"
// → Agent 自动转换为 DSL → 查询 Being → 返回结果
```

**核心卖点**：
- **实时同步**：WMS 库位与 ERP 库存、MES 工单实时一致
- **AGV 联动**：入库/出库/盘点自动触发 AGV 调度
- **语音/自然语言操作**：仓库管理员用语音查询库存、下达拣货指令

**定价**：原有 WMS 维护费 ¥8-12 万/年 → SunDAO WMS 模块 ¥35-50 万/年

### 3.6 AGV 调度 → SunDAO 智能物流

**原有功能**：路径规划、任务调度、交通管制、充电管理

**痛点**：
- 与 WMS/MES 数据不通，调度靠人工干预
- 无法根据生产计划自动调整优先级
- 故障响应慢，影响产线

**SunDAO 升级方案**：

```
def AGV {
    schema { id, status, battery, current_task, location }
    // AGV 作为 Being，状态实时存储在 SunDaoQL
    // MES 工单优先级变化 → 自动重新调度 AGV
}

def TransportTask {
    schema { from_location, to_location, priority, material_id }
    // 任务 Being 与 MES 工单、WMS 出库单自动关联
}

// 全局优化：Agent 根据所有工单、库存、AGV 状态生成最优调度
client_script {
    async fn optimize_fleet() {
        let tasks = await $dao.query().scanByDef("TransportTask")
            .filter({ status: "PENDING" })
            .execute()
        let agvs = await $dao.query().scanByDef("AGV")
            .filter({ status: "IDLE" })
            .execute()
        // 调用 OR-Tools 求解器
        let schedule = await $server.call({
            method: "solver_agv_schedule",
            params: { tasks, agvs }
        })
        // 自动下发任务
        for (const task of schedule) {
            $server.call({ method: "assign_task", params: task })
        }
    }
}
```

**核心卖点**：
- **全局优化**：不是单个 AGV 的路径规划，而是基于生产计划的全局最优调度
- **故障自愈**：AGV 故障 → 自动重新分配任务 → 通知维护人员
- **数字孪生**：AGV 实时状态存储为 Being，支持仿真和预测

**定价**：原有 AGV 维护费 ¥5-10 万/年 → SunDAO AGV 模块 ¥25-40 万/年

## 四、分阶段实施计划

### 4.1 阶段一：试点验证（Y1，2026 Q4 - 2027 Q2）

**目标**：3-5 家 S 级客户签署 POC 合同，验证 SunDaoQL 在真实生产环境的稳定性

**策略**：
1. **选择标准**：
   - 与团队关系最紧密的 3-5 家老客户
   - IT 负责人愿意尝试新技术
   - 有明确的痛点（如追溯困难、数据不通）
   - 愿意作为标杆案例（给予价格优惠）

2. **POC 范围**：
   - 不全面替换，先选一个模块（建议从 **MES 或 WMS** 开始）
   - POC 周期：3 个月
   - POC 目标：证明 SunDaoQL 的性能和稳定性不弱于原有系统

3. **收入预期**：
   - POC 合同单价：¥15-30 万（含部署、培训、3 个月支持）
   - Y1 POC 收入：¥60-100 万

### 4.2 阶段二：模块替换（Y2，2027 Q3 - 2028 Q2）

**目标**：8-10 家客户完成 1-2 个模块的正式替换

**策略**：
1. **基于 POC 成功案例推广**：用标杆客户的实际数据（性能提升、成本降低）说服其他客户
2. **模块优先级**：
   - 第一梯队：MES（痛点最突出，ROI 最容易量化）
   - 第二梯队：WMS（与 MES 联动效果好）
   - 第三梯队：PLM（BOM 变更是核心痛点）
3. **渐进替换**：客户可以先替换 1 个模块，验证后再扩展

4. **收入预期**：
   - 模块替换合同单价：¥50-100 万（含数据迁移、定制开发、1 年维护）
   - Y2 模块替换收入：¥500-800 万
   - 维护费收入：¥100-150 万

### 4.3 阶段三：全面替换（Y3，2028 Q3 - 2029 Q2）

**目标**：15-20 家客户完成 3 个以上模块的替换，部分客户全面替换

**策略**：
1. **推出"智能制造平台"打包方案**：
   - 报价 + PLM + ERP + MES + WMS + AGV 全链路
   - 打包价 ¥150-250 万（比单买便宜 30%）
   - 包含 AI Agent（智能排程、质检辅助、预测维护）

2. **行业模板加速**：
   - 汽车零部件行业模板
   - 电子制造行业模板
   - 新能源电池行业模板
   - 新客户基于模板快速上线（从 6 个月缩短至 6 周）

3. **收入预期**：
   - 全面替换合同单价：¥150-250 万
   - Y3 全面替换收入：¥1,500-2,000 万
   - 维护费 + 升级收入：¥500-800 万

### 4.4 阶段四：平台生态（Y4+，2029 后）

**目标**：从"项目制"转向"平台订阅制"

**策略**：
1. **SaaS 托管服务**：客户无需本地部署，使用 SunDAO Cloud
2. **组件市场**：第三方开发者为 SunDaoApp 开发行业组件

## 六、关键成功因素与风险

### 6.1 关键成功因素

| 因素 | 具体措施 | 负责人 |
|------|---------|--------|
| **产品稳定性** | POC 前完成核心引擎压力测试 | CTO |
| **数据迁移工具** | 开发原有系统→SunDAO 的自动化迁移工具 | 存储引擎团队 |
| **销售培训** | 现有销售团队培训 SunDAO 价值主张 | CMO |
| **标杆案例包装** | 每个 POC 客户产出案例研究（ROI 量化） | 市场经理 |
| **客户成功** | 专属客户成功经理，确保 POC 成功 | 销售总监 |

### 6.2 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| POC 失败 | 中 | 高 | 严格筛选 POC 客户；缩小 POC 范围；预留技术攻坚时间 |
| 数据迁移困难 | 中 | 高 | 开发自动化迁移工具；提供数据清洗服务；分阶段迁移 |
| 客户接受度低 | 低 | 高 | 利用现有客户关系；用实际 ROI 数据说服 |
| 原有产品维护收入下降 | 高 | 中 | 维护费逐步转为 SunDAO 订阅费；老客户给予升级优惠 |
